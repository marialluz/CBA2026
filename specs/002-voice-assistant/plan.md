# Implementation Plan: Renault Voice Assistant

**Branch**: `002-voice-assistant` | **Date**: 2026-03-30 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/-voice-assistant/spec.md`

## Summary

Build a Flutter mobile app that lets Renault owners have
real-time voice conversations with an ElevenLabs Conversational AI
Agent. The agent answers questions about the vehicle using the owner
manual as its knowledge base, speaking in Brazilian Portuguese. After
each call, the user can view, copy, and share a full transcript. The
app connects directly to ElevenLabs via the official Flutter SDK
(`elevenlabs_agents`) with no custom backend.

## Technical Context

**Language/Version**: Dart (latest stable) with Flutter SDK
**Primary Dependencies**: `elevenlabs_agents` ^0.4.0, `flutter_riverpod` ^2.x, `drift` ^2.x, `permission_handler` ^11.x
**Storage**: SQLite via `drift` (local on-device, two tables: call_sessions, conversation_turns)
**Testing**: `flutter_test` (widget + unit), `mockito` for mocking ElevenLabs client
**Target Platform**: iOS 15+ and Android API 26+ (cross-platform via Flutter)
**Project Type**: mobile-app
**Performance Goals**: Call connection within 5 seconds, UI state feedback within 500ms
**Constraints**: Real-time audio streaming requires stable internet, no offline capability
**Scale/Scope**: Single-user app, ~4 screens (home, call, transcript, permission prompt), ~15 widgets

| III. Observability | PASS | Structured logging via `logging` package for all call events (start, turn, error, end) with session ID and timestamps. Severity levels enforced. No sensitive audio data logged. |

| V. Cross-Platform Consistency | PASS | Flutter ensures identical core flows. Platform-specific code limited to permissions (via `permission_handler`) and audio config (via SDK internals). Tested on both iOS and Android. |

**Structure Decision**: Standard Flutter single-project layout. No
separate backend or API project (spec constraint: no custom backend).
All code lives under `lib/` with `test/` mirroring the structure.
Feature modules are flat (screens, services, providers) rather than
feature-grouped, because the app has only one feature domain (voice
calls + transcripts) — a feature-folder pattern would be premature
abstraction.
